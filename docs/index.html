Summary

We will build an in-memory LSM-Tree key-value store and parallelize it across three levels: GPU-accelerated compaction via CUDA merge path, multi-threaded concurrent operations via OpenMP, and SIMD-accelerated key comparisons via AVX2. By keeping all data in memory, we eliminate disk I/O variability and isolate the computational parallelism of a real storage engine's critical path. Our primary goal is to characterize the speedup contribution of each parallelization technique under different workload patterns.

Background

What is an LSM-Tree?

An LSM-Tree (Log-Structured Merge Tree) is a write-optimized key-value store used in systems like RocksDB, LevelDB, and Cassandra. It converts all writes into sequential sorted batches, achieving high write throughput at the cost of slightly slower reads.

Write path: All put(key, value) operations insert into a sorted in-memory structure (the active memtable). When the memtable reaches a size threshold, it freezes into an immutable memtable, a new active memtable is created, and a background thread flushes the immutable memtable as a sorted array ("SSTable") into Level 0.

Compaction: SSTables accumulate across levels (L0, L1, L2, ...), where each level's capacity is ~10x the previous. When a level exceeds its limit, selected SSTables are merged with overlapping SSTables in the next level via sorted merge. This is the most compute-intensive operation.

Read path: A get(key) searches the active memtable, then the immutable memtable, then L0 SSTables (newest first), then deeper levels via binary search. The first match found is the most recent value.

Why is this suitable for parallelism?

Compaction is essentially a multi-way merge of large sorted arrays — a highly data-parallel operation. Reads are independent across levels and across requests. Flush, compaction, and request serving can be pipelined across CPU cores and GPU simultaneously. The immutability of SSTables means most reads are conflict-free.

In-Memory Design

Our implementation keeps all data in memory: SSTables are std::vector<KeyValue> rather than on-disk files. "Flush" copies sorted data from memtable to a vector; "compaction" merges vectors. This eliminates disk I/O as a confounding variable and lets us measure pure computational parallelism.

The Challenge

Workload characteristics: Compaction involves merging large sorted arrays. Serial merge is inherently sequential (each output depends on the previous comparison), requiring algorithmic restructuring (merge path) to parallelize. Memory access patterns differ across operations: compaction is sequential (good locality), point lookups are random (poor locality), and concurrent operations contend for shared metadata.

System-level concurrency: Background compaction modifies the level structure (replacing old SSTables with merged ones) while foreground threads read from those same SSTables. We address this with version-based snapshot isolation: each reader captures an atomic snapshot of the current level layout before querying. Compaction atomically swaps in a new version via shared_ptr + atomic_store. Old versions remain valid until all readers release them.

GPU offloading tradeoffs: For small SSTables, host-device transfer time dominates kernel execution time, making GPU slower than CPU. Characterizing this crossover point and implementing adaptive offloading is a key challenge.

Load imbalance: In leveled compaction, the amount of work per compaction job varies significantly depending on key range overlap between levels. Naive static assignment may leave some threads idle while others are overloaded.

Goals and Deliverables

Plan to Achieve (MVP)

  1. Serial baseline: Complete in-memory LSM-Tree with put, get, scan, leveled compaction, and correctness tests.
  2. CUDA merge path: GPU-accelerated two-way sorted merge for compaction. Micro-benchmark comparing serial vs CUDA merge across input sizes (100K–50M entries), with time breakdown into transfer, kernel, and transfer back.
  3. OpenMP concurrent operations: Thread pool serving parallel get/put requests via work queue. Pipelined flush and compaction in background threads.
  4. Version-based concurrency control: Atomic version snapshot using shared_ptr + atomic_store, ensuring readers are never blocked by compaction.
  5. End-to-end evaluation: YCSB-style workloads (write-heavy, read-heavy, mixed) comparing serial vs parallel throughput. Speedup analysis with Amdahl's Law breakdown.

Hope to Achieve (Stretch Goals)

  1. Adaptive CPU/GPU compaction: runtime offloading decision based on SSTable size.
  2. SIMD key comparison (AVX2) in merge inner loop and binary search.
  3. Multi-way merge on GPU for L0 compaction (k-way via CUDA streams).
  4. Software prefetching for level traversal and merge operations.
  5. CUDA streams for overlapping data transfer with kernel execution.

Fallback Plan

If GPU merge path proves too complex, we fall back to OpenMP-only parallelism: concurrent reads, CPU-parallel merge using OpenMP threads with binary-search partitioning, and pipelined flush. This still covers multiple 15-418 topics (work distribution, synchronization, locality) and provides meaningful speedup analysis.

Platform Choice

GHC machines provide the combination of multi-core CPU, NVIDIA GPU, and AVX2 support needed to explore all three levels of parallelism. We chose C++ for low-level control over memory layout and SIMD intrinsics. We use OpenMP over MPI because our workload is shared-memory, and CUDA for GPU compute with nvprof/nsight for profiling.
