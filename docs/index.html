Summary

We will build an in-memory LSM-Tree key-value store using Size-Tiered Compaction and parallelize it at two primary levels: GPU-accelerated compaction via CUDA merge path (two-phase shared memory algorithm) with chunk-based pipelining using Co-rank partitioning, and multi-threaded concurrent operations via OpenMP task scheduling. Size-Tiered compaction produces exponentially growing SSTables (multi-GB at deeper levels), perfectly matching the high-throughput, bulk-synchronous nature of GPUs. Cascaded 2-way merges across CUDA streams break the K-way merge serial dependency. By keeping all data in pre-allocated memory pools, we eliminate I/O and allocation noise to isolate pure computational parallelism. Our primary goal is to characterize the speedup contribution of each parallelization technique under different workload patterns.

Background

What is an LSM-Tree?

An LSM-Tree (Log-Structured Merge Tree) is a write-optimized key-value store used in systems like RocksDB, LevelDB, and Cassandra. It converts all writes into sequential sorted batches, achieving high write throughput at the cost of slightly slower reads.

Write path: All put(key, value) operations insert into a sorted in-memory structure (the active memtable). When the memtable reaches a size threshold, it freezes into an immutable memtable, a new active memtable is created, and a background thread flushes the immutable memtable as a sorted array ("SSTable") into Level 0.

Size-Tiered Compaction: We adopt Size-Tiered compaction (as in Cassandra) over Leveled compaction (as in RocksDB). Each level accumulates N similarly-sized SSTables (e.g., 4); when a level is full, all SSTables are merged into a single larger SSTable at the next level. SSTable sizes grow exponentially — Level 2 can reach several GB. This choice inherently generates exponentially growing, large-scale sequential merge tasks, perfectly matching the high-throughput, bulk-synchronous processing nature of GPUs. By contrast, Leveled compaction's small, fragmented merge jobs (tens of MB each) would be dominated by PCIe transfer latency, negating any GPU advantage.

Cascaded 2-way merge: Size-Tiered compaction naturally requires K-way merge (e.g., merging 4 SSTables). A naive sequential K-way merge creates a long serial dependency chain: merge A+B, then AB+C, then ABC+D. Instead, we decompose it into a cascaded binary merge tree that breaks this serial dependency: merge A+B on CUDA Stream 0 and C+D on Stream 1 in parallel, then merge the two results AB+CD. This reduces the critical path from 3 sequential merges to 2 levels (log2K), and reuses a single well-optimized 2-way merge path kernel throughout.

Read path: A get(key) searches the active memtable, then the immutable memtable, then L0 SSTables (newest first), then deeper levels via binary search. The first match found is the most recent value.

GPU Merge Path Algorithm

Our CUDA merge kernel uses the Merge Path algorithm [1] with a two-phase approach to maximize GPU memory throughput:

Phase 1 — Coarse-grained partitioning (Global Memory): Each CUDA block is responsible for producing a fixed-size segment (e.g., 1024 elements) of the merged output. Block i's first thread performs a binary search along the i-th cross-diagonal of the merge matrix to find the partition boundary — the exact (ai, bi) positions in both input arrays where this block's work begins. This is O(log N) per block.

Phase 2 — Fine-grained merging (Shared Memory): All threads in the block cooperatively load their input segments from Global Memory into Shared Memory (coalesced reads). Each thread then performs a local merge path binary search within Shared Memory to determine its individual output elements, and writes the merged result back to Shared Memory. Finally, all threads cooperatively write the sorted segment back to Global Memory (coalesced writes). Operating in Shared Memory avoids the latency of repeated Global Memory accesses and enables coalesced access patterns throughout.

Why is this suitable for parallelism?

Compaction is essentially a multi-way merge of large sorted arrays — a highly data-parallel operation. With Size-Tiered compaction, the merge inputs at deeper levels reach GB scale, providing enough data to saturate GPU compute and amortize PCIe transfer costs. Reads are independent across levels and across requests. Flush, compaction, and request serving can be pipelined across CPU cores and GPU simultaneously. The immutability of SSTables means most reads are conflict-free.

In-Memory Design

Our implementation keeps all data in memory: SSTables are stored in pre-allocated memory pools rather than on-disk files. "Flush" copies sorted data from memtable into a pool buffer; "compaction" merges buffers. GPU-side memory is also pre-allocated at initialization to avoid costly cudaMalloc during runtime. This eliminates both disk I/O variability and memory allocation overhead, letting us measure pure computational parallelism.

The Challenge

Workload characteristics: Compaction involves merging large sorted arrays. Serial merge is inherently sequential (each output depends on the previous comparison), requiring algorithmic restructuring (merge path) to parallelize. Memory access patterns differ across operations: compaction is sequential (good locality), point lookups are random (poor locality), and concurrent operations contend for shared metadata.

PCIe transfer bottleneck: On RTX 2080 (PCIe Gen3 x16, ~15.75 GB/s theoretical), transferring a 1 GB merge job to GPU and back takes ~127 ms in transfer time alone, while the merge kernel may finish in single-digit ms. Naive "send all, compute, receive all" will be slower than CPU. We address this with chunk-based pipelining: the CPU first performs a fast O(log N) Co-rank binary search (merge path partition) to determine, for each fixed-size output chunk, the variable-sized input boundaries in both sorted arrays. Then, multiple CUDA streams concurrently handle the variable-sized Host-to-Device input transfers, kernel executions, and fixed-sized Device-to-Host output transfers, all using pinned memory (cudaMallocHost) to enable cudaMemcpyAsync. This overlaps transfer and compute to hide PCIe latency.

System-level concurrency: Background compaction modifies the level structure (replacing old SSTables with merged ones) while foreground threads read from those same SSTables. We address this with version-based snapshot isolation: each reader captures an atomic snapshot of the current level layout before querying. Compaction atomically swaps in a new version via shared_ptr + atomic_store. Old versions remain valid until all readers release them. Note: concurrent shared_ptr read/write requires std::atomic<std::shared_ptr> (C++20) to be safe; we will profile reference-count contention and consider epoch-based reclamation (EBR) if it becomes a bottleneck.

GPU offloading tradeoffs: For small SSTables, host-device transfer time dominates kernel execution time, making GPU slower than CPU even with pipelining. Characterizing this crossover point and implementing adaptive offloading is a key challenge.

Memory allocation overhead: Frequent allocation of multi-GB buffers for merge outputs causes OS page faults and cudaMalloc overhead that can mask parallelism gains. We use pre-allocated memory pools on both host and device, reusing buffers across compaction jobs.

Goals and Deliverables

Plan to Achieve (MVP)

  1. Serial baseline: Complete in-memory LSM-Tree with put, get, scan, size-tiered compaction, memory pools, and correctness tests.
  2. CUDA merge path with pipelining: GPU-accelerated two-way sorted merge using the two-phase Merge Path algorithm (coarse-grained partitioning in Global Memory, fine-grained merging in Shared Memory). Chunk-based pipeline: CPU performs Co-rank partitioning to determine variable-sized input boundaries, then multiple CUDA streams overlap H2D transfer, kernel execution, and D2H transfer using pinned memory (cudaMallocHost). Pre-allocated device memory pool. Micro-benchmark comparing serial vs CUDA merge across input sizes (1M–50M entries), with time breakdown into transfer, kernel, and overlap efficiency.
  3. Cascaded 2-way merge: Decompose size-tiered K-way compaction into parallel 2-way merges across CUDA streams (e.g., A+B on Stream 0, C+D on Stream 1, then AB+CD).
  4. OpenMP concurrent operations: Thread pool with dynamic task scheduling (not static core pinning) serving parallel get/put requests. Background flush and compaction dispatched as OpenMP tasks, allowing the runtime to balance load across cores based on queue depth.
  5. Version-based concurrency control: Atomic version snapshot using std::atomic<std::shared_ptr> (C++20), ensuring readers are never blocked by compaction.
  6. End-to-end evaluation: YCSB-style workloads (write-heavy, read-heavy, mixed) comparing serial vs parallel throughput. Speedup analysis with Amdahl's Law breakdown.

Hope to Achieve (Stretch Goals)

  1. Adaptive CPU/GPU compaction: runtime offloading decision based on SSTable size, with CPU-GPU co-execution for large compactions (e.g., 80% GPU + 20% CPU OpenMP merge in parallel).
  2. SIMD key comparison (AVX2) with Eytzinger layout for cache-friendly binary search, or N-ary search using AVX2 to compare 4 keys per SIMD instruction.
  3. Epoch-based reclamation (EBR) replacing shared_ptr reference counting to eliminate cache-line contention under high concurrency.
  4. Sharded memtable: partition active memtable by key hash across per-thread buffers for lock-free parallel writes.

Fallback Plan

If GPU merge path with pipelining proves too complex, we fall back to OpenMP-only parallelism: concurrent reads, CPU-parallel merge using OpenMP threads with merge-path partitioning, and pipelined flush. This still covers multiple 15-418 topics (work distribution, synchronization, locality, memory hierarchy) and provides meaningful speedup analysis.

Platform Choice

GHC machines provide the combination of multi-core CPU, NVIDIA GPU, and AVX2 support needed to explore all three levels of parallelism. We chose C++ for low-level control over memory layout and SIMD intrinsics. We use OpenMP over MPI because our workload is shared-memory, and CUDA for GPU compute with nvprof/nsight for profiling.
